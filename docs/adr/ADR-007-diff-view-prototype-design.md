# ADR-007: Diff View 試作方針

- **ステータス**: 決定済み
- **決定日**: 2026-03-25
- **決定者**: yasuabe2613

---

## コンテキスト

フェーズ0では、DAG view と同様に Diff view もアプリ本体から独立して先行試作する。

この段階で次の判断が必要だった。

1. Diff view のフェーズ0スコープをどこまでにするか
2. diff 描画ライブラリとして何を採用するか
3. ダミーデータの入力形式をどうするか
4. commit diff と WIP diff をフェーズ0でどう扱うか

`SPEC.md` では diff 表示に「unified diff」「旧行番号＋新行番号の2列」「シンタックスハイライト」が求められている。一方で、フェーズ0では右ペインやファイル一覧との連携まで含めると責務が広がりすぎる。

---

## 決定

### 1. フェーズ0の Diff view は本文コンポーネントのみを対象とする

フェーズ0で試作するのは、選択済みの単一ファイルについて diff 本文を描画する `diff-view` コンポーネントのみとする。

次はスコープ外とする。

- ファイル一覧
- ファイル選択 UI
- 右ペインとの連携
- 複数ファイル diff の同時表示

### 2. フェーズ0の diff 描画には `diff2html` を採用する

フェーズ0では `monaco-editor` ではなく `diff2html` を使用する。

### 3. フェーズ0の入力は `raw unified diff string` とする

`diff-view` コンポーネントは、パース済み構造ではなく unified diff 文字列をそのまま受け取る。

ダミーデータも同じ形式で保持する。

### 4. commit diff と WIP diff は同一コンポーネントで扱う

フェーズ0では diff 本文しか扱わないため、commit diff と WIP diff の違いは入力文字列の違いに留める。

表示コンポーネントは区別しない。

---

## 理由

### スコープを狭く保つため

フェーズ0の目的は技術的に難しい部品の先行検証である。Diff view では、まず本文描画の品質とライブラリ適合性を確認すれば十分であり、ファイル一覧や選択状態の設計は後続フェーズへ分離した方がよい。

### `diff2html` が用途に直接合うため

今回必要なのは編集機能ではなく閲覧専用の diff 表示である。`diff2html` は unified diff 文字列から行番号付き HTML diff を直接生成でき、フェーズ0で必要な表示要件を最短で満たせる。

一方、`monaco-editor` は高機能だが、フェーズ0で必要な範囲に対しては重く、導入時の設計判断も増える。

### 実データへの接続を簡単にするため

将来的に git から取得する diff も本質的には unified diff 文字列である。フェーズ0から入力形式をそれに揃えておくことで、ダミーデータから実データへの差し替えが単純になる。

### WIP と commit の差を過剰に持ち込まないため

フェーズ0では本文描画だけが関心事であり、WIP / commit の違いは UI ではなくデータ取得側の責務に近い。表示コンポーネントを共通化する方が設計が素直である。

---

## 影響・トレードオフ

| 観点 | 内容 |
|---|---|
| 実装速度 | `diff2html` と raw diff 入力により、フェーズ0を素早く立ち上げられる |
| 表示の自由度 | `diff2html` の既定デザインに一部制約される |
| スコープ管理 | 右ペインやファイル一覧を切り離せるため、本文描画の検証に集中できる |
| 将来の拡張 | 複数ファイル表示やファイル一覧連携はフェーズ1以降に別途設計が必要 |
| WIP / commit 共通化 | 表示コンポーネントは単純になるが、取得元の差異は別レイヤで吸収する前提になる |
