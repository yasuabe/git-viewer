# ADR-003: データモデルとアーキテクチャ方針

- **ステータス**: 決定済み
- **決定日**: 2026-03-23
- **決定者**: yasuabe2613

---

## コンテキスト

コミットグラフ表示の実装にあたり、以下の3点について設計方針を決定する必要があった。

1. コミットの順序をどこで決めるか
2. レーン情報をどの層に持たせるか
3. インクリメンタル更新をどう扱うか

---

## 決定1: コミットの順序は `git log` に委ねる

`simple-git` 経由で `git log --topo-order --date-order` を使用し、アプリ側での再ソートは行わない。

### 理由

- `--topo-order`（トポロジー順保証）と `--date-order`（同じ深さなら日付順）の組み合わせは、ADR-001 で言及した Temporal Topological Sort とほぼ等価な結果を得られる
- git 本体が C で最適化済みであり、自前実装より高速かつ堅牢
- clock skew（非同期クロック由来の日付異常）等のエッジケースを git 本体が長年かけて対処済み
- `git log` の表示順に従うことで、ユーザーの期待と一致しやすい

### トレードオフ

意図的に過去日付を設定したコミット等、`--topo-order --date-order` でも防げないエッジケースは残る。ただし GitKraken も同様に許容しているケースであり、許容範囲とする。

## 決定2: 3層アーキテクチャとレーン情報の所有

アプリ全体を制御層・モデル層・プレゼンテーション層の3層で構成する。ブランチ情報は両層に持たせ、モデル層は生データ、プレゼンテーション層は加工後データを保持する。

### 各層の責務

**制御層**
- リポジトリの監視（`.git/HEAD` 等の変更検知）
- `simple-git` の呼び出し
- config データの取得
- ユーザ操作のハンドリング

**モデル層**
- コミットデータ（hash・author・date・message・parents）
- ブランチ情報（生データ）: ブランチ名→コミット hash のマッピング、HEAD の指し先、local / remote / worktree の区別

**プレゼンテーション層**
- モデル層のコミットデータを監視し、レーンデータを計算・保持
- ブランチ情報（加工後）: ブランチの生存区間（行範囲）・列番号・色
- Canvas 2D による再描画

### 理由

- モデル層を git の概念に忠実に保つことで、制御層との境界が明確になる
- レーンは git の概念ではなく表示上の概念であるため、プレゼンテーション層が所有するのが自然
- ブランチ情報を両層に持たせることで、モデル層は生データの純粋な保持に専念し、プレゼンテーション層は描画都合に特化できる

---

## 決定3: 更新時は全再計算とする

リポジトリ更新（`git fetch` 等）やリロード時は、グラフ全体を差分なしで再計算する。

### 理由

- 閲覧専用ツールという性格上、更新頻度が低く全再計算のコストが問題になりにくい
- 差分更新はレーン割り当ての整合性維持が複雑になり、実装リスクが高い
- シンプルな実装を優先する

### トレードオフ

数万コミット規模のリポジトリで更新時に一瞬の遅延が生じる可能性がある。パフォーマンス問題が顕在化した場合は差分更新への移行を検討する。
